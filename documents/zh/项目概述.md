# 项目概述 - CS6290 小组项目：Polymarket 信号分析

**日期**：2026-02-13

**状态**：**初版** 

## 1. 项目概述 (Project Overview)

### 核心选题

Polymarket Signal Analysis — Empirical Detection of Unusual Odds Movements.

### 核心形态

一个基于 Web 的**静态分析展示页 (Static Analytical Showcase)**。 项目组将在后台离线运行数据管道，生成特定市场的"异常事件"数据集。最终交付物是一个无需后端实时支持的网页，用于交互式地展示检测结果和统计佐证。

## 2. 目标与非目标 (Goals & Non-Goals)

### 2.1 目标 (Goals)

- **核心目标**：实现一套离线算法流程，能够输入历史数据，输出"异常事件"列表。
- **展示目标**：构建一个前端界面，清晰地将"赔率异常"与"统计佐证"（如 Z-Score 突变、成交量激增）对应展示。
- **工程目标**：
  - 实现数据清洗与重采样（Resampling）机制。
  - 实现两类特定的异常检测算法，**展示其因果关系**：
    - **原因（微观层面）：狙击手攻击检测**。通过识别特定的恶意钱包行为来追踪根本原因。
    - **症状（宏观层面）：市场层面统计异常检测**。检测由狙击手行为导致的可观察市场干扰。

### 2.2 非目标 (Non-Goals)

- **不做实时**：演示时不连接 Polymarket 实时 API，不进行实时监控。
- **不做交易**：不提供买卖建议，不连接钱包。
- **不做逐笔分析**：不处理 Tick-level（逐笔）交易数据，不追踪具体账户（Whale Tracking），仅关注聚合后的价格与成交量。
- **不做复杂归因**：不尝试解释"为什么"发生异常（如不抓取外部新闻），只展示"发生了什么"。

## 3. 核心展示策略 (Core Display Strategy)

鉴于展示页的静态性质和浏览器性能限制，我们将在 Streamlit 中采用**"侦探报告"卡片式 UI（钻取交互）**来直观展示两个异常维度之间的因果联系：

- **第 1 级：宏观证据**：卡片初始显示宏观层面症状（例如分钟级交易量激增或极端价格 Z-Score）。
- **第 2 级：微观追踪**：用户可以展开卡片以揭示微观层面的根本原因，显示特定的可疑钱包地址、其孤立森林异常分数以及狙击手攻击的确切序列（例如大额快速买卖对）。

## 4. 核心项目流程 (Project Pipeline & User Flows)

本项目的数据流向被严格划分为三个阶段：

### 阶段一：数据获取与预处理 (Data Acquisition & Preprocessing)

*目标：构建标准化的、时间对齐的历史数据集。*

1. **获取 (Ingest)**:
   - 脚本调用 Polymarket API (CLOB/History) 拉取指定市场（如 "US Election 2024"）过去 6 个月的原始 OHLCV 数据。
   - 获取互斥合约对的数据（例如同时获取 "Yes" 和 "No" 的价格序列）。
2. **清洗与对齐 (Clean & Align)**:
   - **填补缺失值**: 处理 API 返回的空洞。
   - **时间对齐 (Time Alignment)**: 这一点至关重要。必须确保 "Yes" 和 "No" 的价格数据在相同的时间戳上对齐（Resampling 到统一的分钟/小时级），以便后续计算平价公式。

### 阶段二：算法检测 (Algorithmic Detection)

*目标：运行离线算法，输出异常事件列表。我们将重点检测两类模式：*

1. **模式 A：市场层面统计异常检测**
   - **定义**: 通过对聚合价格和成交量数据的统计分析来检测异常市场行为。
   - **检测逻辑**:
     - 计算价格的移动平均线 (MA) 和标准差 (StdDev)。
     - 识别 **Volume Spikes** (成交量激增) 伴随 **High Z-Score** (价格偏离均值 > 3σ) 的时间窗口。
   - **输出**: 标记为 "Type A: Market Anomaly"。
2. **模式 B：狙击手攻击检测**
   - **定义**: 检测操纵行为，攻击者使用大额资金拉高价格，在散户跟风后快速卖出。
   - **检测逻辑**:
     - 使用基于会话的分析（同一地址 30 分钟内的交易）。
     - 应用孤立森林算法和 AI 特征（Sentence-BERT 用于交易序列）。
     - 识别最异常的 5% 会话，具有特定的狙击手特征。
   - **输出**: 标记为 "Type B: Sniper Attack"。

### 阶段三：结果展示 (Result Showcase)

*目标：用户在 Web 界面查看阶段二生成的分析结果。*

1. **全局概览 (Global Overview)**:
   - 用户打开网页，看到全量时间轴的赔率走势。
   - 时间轴上用不同颜色标记了 **Type A (市场异常)** 和 **Type B (狙击手攻击)** 异常点。
2. **交互下钻 (Interactive Drill-down)**:
   - 用户点击一个 **Type A** 标记 -> 弹窗展示该时刻的成交量柱状图和波动率曲线（证明统计异常）。
   - 用户点击一个 **Type B** 标记 -> 弹窗展示交易序列和会话分析（证明狙击手攻击模式）。

## 5. 关键决策与风险管理 (Key Decisions & Risk Management)

### 5.1 已识别的风险 (Identified Risks) & 缓解策略 (Mitigation Strategies)

| 风险点                             | 描述                                                         | 缓解策略 (Mitigation)                                        |
| ---------------------------------- | ------------------------------------------------------------ | ------------------------------------------------------------ |
| **数据不可用 (Data Unavailability)** | 狙击手攻击检测需要钱包地址信息，当前数据源可能不包含这些信息。 | **数据源扩展**。从 Polymarket API 或区块链浏览器获取包含钱包地址的原始交易数据，以支持基于会话的分析。 |
| **浏览器性能瓶颈 (Browser Performance Bottleneck)** | 6个月的高频数据直接加载会让浏览器崩溃。                      | **按需加载**。前端默认只加载"小时级"概览数据；只有用户点击特定事件时，才加载该片段的"分钟级"详情。 |
| **算法"事后诸葛亮" (Algorithm "Hindsight Bias")** | 预计算结果可能看起来像是人工挑选的。                         | 在界面上**明确展示算法阈值**（如："检测阈值：Spread > 0.05"），表明这是基于规则的客观筛选。 |

### 5.2 技术栈暂定 (Tentative Tech Stack)

- **数据处理**: Python (Pandas, NumPy, Scikit-learn)
- **数据源**: Polymarket Clob API / Historical Data Snapshots
- **前端展示**: Streamlit
- **数据格式**: Static JSON files
